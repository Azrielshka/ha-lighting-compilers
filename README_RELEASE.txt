# HA Lighting Compilers — инструкция

Инструмент собирает конфигурацию Home Assistant для DALI-освещения из одной
проектной Excel-таблицы и, отдельным действием, доставляет её на объект.

Версия сборки видна в шапке окна. Называйте её, когда с чем-то обращаетесь.

---

## Что это за штука и чего от неё ждать

На вход — Excel с адресами ламп, датчиков и панелей. На выходе — YAML для
Home Assistant: группы света, пространства, вспомогательные объекты, скрипты,
автоматизации и карточки дашборда.

```
Excel  →  проверка  →  нормализация  →  генераторы  →  YAML  →  деплой на HA
```

Два свойства, которые стоит понимать сразу:

**Генерация полностью офлайновая.** Чтобы собрать конфигурацию, Home Assistant
не нужен вообще — ни сети, ни объекта. Можно спокойно готовить всё на ноутбуке
дома. В живую систему пишет только шаг деплоя, по явному нажатию.

**`launcher.exe` НЕ содержит генераторы внутри.** Он запускает `scripts/*.py`
через Python из `.venv`. Отсюда два следствия: Python нужен обязательно, а
структуру папок менять нельзя.

---

## Что внутри архива

```
launcher.exe                окно, из которого всё запускается
scripts/                    генераторы. ОБЯЗАТЕЛЬНЫ
templates/                  шаблоны карточек и скриптов — можно править
data/                       сюда кладёте свой Excel, сюда же ложится результат
  object_example.xlsx       рабочий пример: на нём можно проверить, что всё встало
  example.xlsx              старый пример v1, на нынешнем пайплайне НЕ работает
requirements.txt            зависимости Python
bash_command.txt            шпаргалка команд, если пойдёте в консоль
env.example                 образец .env для деплоя из консоли
INSTALL.txt                 этот файл
THIRD-PARTY-LICENSES.md     лицензии стороннего материала
```

## Установка

### 1. Python

Нужен **Python 3.10–3.12** с python.org. При установке — галочка
**Add Python to PATH**.

Проверка:

```
py --version
```

Если команда не найдена — Python не в PATH. Переустановите с галочкой либо
добавьте путь руками.

### 2. Распаковать архив

Куда угодно, но **без кириллицы и пробелов в пути** — это избавит от части
проблем с кодировкой. Например:

```
C:\Projects\ha-lighting-compilers
```

### 3. Открыть папку в консоли

В Проводнике откройте папку, наберите в адресной строке `powershell` и Enter.
Консоль откроется уже в нужном каталоге.

### 4. Создать окружение и поставить зависимости

```
py -m venv .venv
.venv\Scripts\python.exe -m pip install -r requirements.txt
```

⚠ Именно `.venv\Scripts\python.exe -m pip`, а **не** голый `pip`. Голый `pip`
ставит пакеты в то окружение, которое первым нашлось в PATH, — обычно в
системное. Установка пройдёт успешно, а лаунчер потом скажет, что модуля нет,
и вы будете искать причину не там.

Ставится около гигабайта: основной вес — PySide6 (оконная библиотека) и
pyarrow. Скачивание займёт несколько минут.

## Проверка, что всё встало

Не переходите к своей таблице, пока это не пройдёт: так вы отделите проблемы
установки от проблем таблицы.

```
.venv\Scripts\python.exe scripts\validate_excel.py --excel data\object_example.xlsx
```

Ожидаемый ответ — вот такой:

```
📊 Статистика таблицы:
  Строк данных:  91
  Помещений:     8
  Групп:         16
  Ламп:          91
  Датчиков:      15
  Панелей:       5
  Единиц обсл.:  5

✅ Таблица прошла валидацию без замечаний
```

Цифры должны совпасть — это тот же файл. Не совпали или вылетела ошибка —
проблема в установке, дальше идти смысла нет.

Дальше запустите окно:

```
launcher.exe
```

В шапке — имя инструмента и версия. Если окно не открылось, а консоль
что-то написала — читайте, что именно: обычно это отсутствующий модуль
(зависимости ушли не в то окружение, см. выше).

## Работа

### Поля вверху окна

| Поле | Что это |
|---|---|
| **Project Root** | папка, куда распаковали архив |
| **Excel File Path** | ваша таблица в `data/` |
| **Дашборд** | `url_path` дашборда на объекте, например `dashboard-tets`. Обязан содержать дефис — требование Home Assistant |
| **Объект** | имя в шапке Главной страницы, например «Колледж Химки» |

⚠ «Дашборд» и «Объект» читает **генерация карточек**, а не деплой. Поменяли имя
— прогоните `[ +] LOVELACE CARDS` заново. Деплой ничего не генерирует, он льёт
на объект готовые файлы с диска, и старое имя уедет как есть.

Поля сохраняются между запусками.

### `[ ▶] BUILD ALL` — девять шагов подряд

```
[01] VALIDATE EXCEL      проверка таблицы. Ошибки — пайплайн стоит
[02] NORMALIZE EXCEL     Excel -> parquet, дальше читается только он
[03] LIGHTS GROUPS       группы света по зонам
[04] GENERAL GROUPS      общая группа на помещение
[05] FLOOR GROUPS        группы этажей и техпомещений
[06] AREAS & FLOORS      задание на пространства и этажи
[07] HELPERS             вспомогательные объекты одним пакетом
[08] SCRIPTS             клоны скриптов по единицам обслуживания
[09] AUTOMATIONS         автоматизации + blueprint'ы
```

Проверка идёт первой намеренно: на битой таблице сборка останавливается сразу,
а не показывает ошибки уже в Home Assistant, где искать их вчетверо дороже.

Каждый шаг можно запустить и отдельно — он работает с тем, что уже нормализовано.
Excel читает только шаг 2.

### `[ +] LOVELACE CARDS` — карточки

В `Build All` **не входит**: карточки нужны не всегда, а для залов и рекреаций
на Home Assistant должны стоять карты `mushroom` и `card_mod` из HACS. Без них
раздел отрисуется с ошибкой.

### `[ ↑] DEPLOY → HOME ASSISTANT` — единственный шаг, который трогает объект

Открывается отдельное окно: галочки «что отправить», параметры SSH и Home
Assistant, кнопки **«Показать план (dry-run)»** и **«Отправить на HA (LIVE)»**.

Порядок на объекте:

1. `[ ▶] BUILD ALL`
2. `[ +] LOVELACE CARDS`
3. **Снимок дашборда** — обязательно перед первым деплоем карточек:
   ```
   .venv\Scripts\python.exe scripts\backup_dashboard.py --url http://IP:8123 --token ТОКЕН
   ```
   Запись конфига дашборда перезаписывает его **целиком**. Снимок — ваш способ
   вернуться, если что-то пойдёт не так.
4. **Dry-run.** Покажет, что именно поедет, ничего не отправляя.
5. **LIVE.**

Что нужно на стороне объекта:

- add-on «Advanced SSH & Web Terminal» с `sftp: true`, `username: root`,
  портом наружу и вашим публичным ключом в `authorized_keys`;
- **токен администратора** Home Assistant: профиль → Security → Long-lived
  access tokens → Create token. Обычному пользователю запись конфига дашборда
  запрещена, и деплой карточек молча не пройдёт.

Проверить канал до объекта заранее:

```
.venv\Scripts\python.exe scripts\check_sftp.py --host IP --user root --key путь\к\ключу
```

⚠ **Рестарт Home Assistant деплой НЕ делает.** YAML приедет, но не применится,
пока не перезапустите вручную. Карточек и пространств это не касается — они
применяются сразу.

## Результат

```
data/
├── normalized/        parquet, промежуточный слой
├── light_groups/      группы света
├── areas/             задание на пространства и этажи
├── helpers/           вспомогательные объекты
├── scripts/           клоны скриптов
├── automations/       автоматизации
├── blueprints/        blueprint'ы
├── lovelace/          views дашборда
└── backups/           снимки дашборда (появится после первого бэкапа)
```

⚠ Не путайте `scripts/` в корне архива и `data/scripts/`. В корне — сами
генераторы, их не трогают. В `data/scripts/` — результат их работы, YAML для
Home Assistant. То же и с `templates/` (шаблоны) против `data/blueprints/`
(готовые blueprint'ы на отправку).

## Если пойдёте в консоль

Всё, что делают кнопки, доступно командами — полный список в
`bash_command.txt`. Пример:

```
.venv\Scripts\python.exe scripts\validate_excel.py --excel data\моя_таблица.xlsx
.venv\Scripts\python.exe scripts\normalize_excel.py --excel data\моя_таблица.xlsx
.venv\Scripts\python.exe scripts\generate_lights_groups.py
```

У любого скрипта есть `--help` со списком флагов.

Чтобы не передавать адрес и токен в каждой команде, скопируйте `env.example` в
файл `.env` и заполните его — `deploy.py` и `backup_dashboard.py` прочитают
оттуда.

⚠ Проводник Windows не даёт создать файл с именем `.env` через переименование.
Сделайте из консоли:

```
copy env.example .env
notepad .env
```

`.env` содержит токен — не кладите его в общий доступ.

## Если что-то пошло не так

**«python.exe не найден»** — не создан `.venv`. Вернитесь к шагу 4 установки.

**Кнопка нажимается, ничего не происходит** — проверьте, что
`.venv\Scripts\python.exe` существует, а Project Root указывает на папку, внутри
которой лежит `scripts/`.

**Окно висит и не отвечает** — так и есть: на время работы шага окно
подвисает, Windows может написать «Не отвечает». Дождитесь. Build All на
крупном объекте занимает до полуминуты. Это известное ограничение, не поломка.

**Валидатор ругается на таблицу** — читайте коды. `E*` блокируют генерацию,
`W*` предупреждают, но пропускают. Отчёт целиком лежит в
`data/validation_report.json`.

**Деплой: «invalid auth»** — токен не тот или не админский.

**Задеплоили карточки, а на дашборде старое** — не прогнали
`[ +] LOVELACE CARDS` после правки полей «Объект»/«Дашборд».

**После деплоя пакет не загрузился, в логе HA «duplicate key»** — на объекте
уже есть помощник с таким же именем, заведённый руками. Home Assistant не
сливает его с нашим пакетом, а роняет пакет целиком. Удалите ручного.

**Свет не включается, хотя всё задеплоено** — см. следующий раздел.

## ⚠ Чего инструмент НЕ делает

**Не собирает JSON Zone Manager.** Это отдельная ручная конфигурация. Без неё
автоматизация вызовет `zone_manager.get_sensor_config`, получит `found: false`,
и свет включаться не будет. Инструмент к этому отношения не имеет — вся
остальная конфигурация при этом на месте и корректна.

**Не перезапускает Home Assistant.**

**Не описывает логику за помощниками.** `input_boolean` пресетов зала и режимов
этажей появятся, но это выключатели: что они делают, описывают ваши
автоматизации. Исключение — `input_number.vacant_delay` и
`input_select.nav_floor_<N>`: их читает сгенерированный код.

**Не трогает то, что вы сделали руками.** Файлы на объекте идут с префиксом
`zm_`, разделы дашборда — с `zm-`. Всё остальное деплой сохраняет как есть.
