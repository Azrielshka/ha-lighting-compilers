# 1. Источник данных

Читается **только лист «Проектная БД»**. Имя листа передаётся явно, а не берётся
по порядку. Остальные листы книги (`ПНР`, `ИНФ контроллеры`, `Глоссарий`,
`Группы соседей`) пайплайном не читаются.

## Колонки

| Колонка | Заполнение | Используется |
|---|---|---|
| `Этаж` | каждая строка | да — для `floor_<n>_all` |
| `Название помещения` | первая строка помещения | да — источник `room_slug` |
| `Тип помещения` | первая строка помещения | да — выбор шаблона карточки |
| `Шина DALI` | каждая строка | нет (справочно) |
| `Группа` | **каждая строка** | да — привязка устройств |
| `Лампа` | строка лампы | да |
| `Addr L` | — | **нет** |
| `Датчик` | см. §3 | да |
| `Addr MS` | — | **нет** |
| `Панель` | см. §3 | да |
| `Addr KP` | — | **нет** |

`Название помещения` и `Тип помещения` протягиваются вниз (ffill) в пределах
помещения. `Группа` **не протягивается** — она уже заполнена на каждой строке.

---

# 2. Правило привязки устройств

> Устройство принадлежит той группе, которая указана **в его собственной строке**.

Это главное отличие от v1, где группа задавалась только на стартовой строке, а
датчик привязывался к группе через `is_group_start`.

Следствия:

- у группы **N ламп** (N ≥ 1);
- у группы **0..N датчиков** (в примере `102_1` имеет два: `1.1.2` и `1.1.3`);
- у группы **0..N панелей**;
- понятия `is_group_start` больше не существует.

---

# 3. Значение `None` и пустая ячейка

Различаются:

- **адрес** (`1.1.3`) — устройство есть;
- **`None`** — устройства по проекту нет;
- **пустая ячейка** — «в этой строке ничего не сказано», нормальное состояние.

Правило проверяется **на уровне помещения**, а не группы:

> В каждом помещении колонка `Датчик` должна содержать хотя бы одно непустое
> значение — адрес или `None`. То же для колонки `Панель`.

Достаточно одной такой строки на помещение. Смешанный случай (часть групп с
адресами, часть без) — норма.

## Терпимость к написанию

Токены `None`, `none`, `нет`, `-`, `–`, `—` (без учёта регистра и пробелов)
нормализуются в единое значение «нет устройства». Всё остальное, что не является
адресом `X.Y.Z`, — ошибка валидации.

## ⚠ Ловушка pandas

Строка `None` входит в список `na_values` по умолчанию, поэтому
`pd.read_excel(...)` молча превращает её в `NaN` — и различие между «устройства
нет» и «ячейка не заполнена» исчезает. Читать таблицу **только** так:

```python
df = book.parse(sheet_name=SHEET_NAME, dtype=object, keep_default_na=False, na_values=[])
```

# 4. Адрес `X.Y.Z`

- `X` — номер этажа;
- `Y` — номер шины DALI (нумерация в пределах этажа);
- `Z` — проектный номер устройства (нумерация в пределах шины).

Уникальность обеспечивается связкой «полный адрес + тип сущности». Адрес `1.1.1`
может одновременно принадлежать лампе, датчику и панели — это разные домены и
разные префиксы.

**Источник правды об этаже и шине — адрес, а не колонки.** Колонки `Этаж` и
`Шина DALI` носят справочный характер, их расхождение с адресом даёт
предупреждение, но не блокирует генерацию.

Единственное исключение: `floor_<n>_all` строится по колонке `Этаж`.

# 5. Канон нейминга сущностей

Сущности создаёт сторонняя интеграция; мы их только предсказываем. Имена
гарантированы заказчиком.

| Объект | Правило | Пример |
|---|---|---|
| Лампа | `light.l_X_Y_Z` | `light.l_1_20_15` |
| Датчик движения | `sensor.ms_X_Y_Z` | `sensor.ms_1_20_3` |
| Датчик освещённости | `sensor.il_X_Y_Z` | `sensor.il_1_20_3` |
| Панель | `event.kp_X_Y_Z` | `event.kp_1_1_1` |
| Группа (зона) | `light.<group_id>` | `light.102_1` |
| Общая группа помещения | `light.<room_slug>_obshchii` | `light.104_metodicheskii_kabinet_obshchii` |

Один адрес датчика порождает **две** сущности: `sensor.ms_*` и `sensor.il_*`.
`il_` появляется у любых датчиков и будет использоваться в автоматизациях.

Группы (`light.<group_id>` и `light.<room_slug>_obshchii`) создаём мы сами —
это единственные сущности, которых без нас в HA не будет.

`room_slug` — транслит русского названия помещения. Это единственное место, где
пайплайн зависит от русского текста: переименование помещения в таблице
переименует сущность в HA.

# 6. Типы помещений

Значения приводятся к нижнему регистру с обрезкой пробелов: `" Korridor "` → `korridor`.

| Ключ | В таблице | Семейство | Принцип управления |
|---|---|---|---|
| `class` | Class | — | поддержание освещённости, сцены, диммирование с панели |
| `korridor` | Korridor | `default` | по датчикам движения с соседями |
| `recreation` | Recreation | `default` | на перемене держит яркость, на уроке — как коридор |
| `hall` | Hall | `hall` | несколько датчиков на одну группу света |
| `zal` | Zal | — | только с панелей, сцены; датчиков нет |
| `special` | Special | `special` | микроволновые датчики, выключение по потере присутствия |

`generic` как тип и как fallback **упразднён**.

`class` и `zal` **не автоматизируются**: у класса поддержание освещённости —
вне области нашего пайплайна, зал управляется только панелями. Скрипты и
автоматизации им не создаются. Датчики класса при этом остаются в parquet
и попадают в карточки.

Помещение без типа или с неизвестным типом:

- **исключается из генератора карточек** (с предупреждением);
- **остаётся** в `devices` / `groups` / группах света и в группах этажа —
  лампы в нём физически существуют.

`Korridor` пишется именно так (не `Corridor`) — это осознанное решение заказчика.

# 6.1 Единицы обслуживания (колонка «Блок»)

## Зачем

Один экземпляр скрипта в Home Assistant — это **одна очередь**. При тысяче
датчиков вызовы копятся, и свет отстаёт от человека; `mode: queued` не спасает,
а усугубляет. Поэтому шаблонные скрипты **клонируются**: у каждой единицы
обслуживания свой набор, и они друг друга не ждут.

## Правило

> **Единица обслуживания** — помещение целиком, если «Блок» пуст.
> Если «Блок» заполнен — все помещения с одинаковым значением.

На каждую единицу: **один набор скриптов + две автоматизации** (ON и OFF).

## Почему колонка, а не эвристика

Правила заказчика звучат так: «лестницы склеиваем по стояку», «санузлы —
с ближайшими на этаже», «длинный коридор режем пополам». Всё это —
**пространственная информация**, которой в таблице нет и вывести её нельзя:
номер помещения не говорит, какая лестница над какой.

Эвристики («лестницы с одинаковой последней цифрой — один стояк») работали бы
на одном объекте и **молча врали** на следующем. Поэтому решение принимает
проектировщик, глядя на план, и записывает его в колонку. Одна колонка вместо
десяти догадок.

## Что клонируется

| Семейство | Blueprint'ы | Скрипты | Всего |
|---|---|---|---|
| `default` | `zm_default_on/off` | `on`, `off`, `near_off` | 3 |
| `hall` | `zm_hall_on/off` | `on`, `off`, `hall_near` | 3 |
| `special` | `zm_special_on/off` | `special_on`, `special_off` | 2 |

## Имена

Из `unit_id` — это «Блок» либо `room_slug` помещения:

```
script.103_vestibiul_on        ← помещение само по себе
script.103_vestibiul_off
script.103_vestibiul_near_off

script.ladder_1_on             ← блок из нескольких помещений
script.ladder_1_off
script.ladder_1_near_off
```

## Предел 12 датчиков

Зашит **в сами blueprint'ы**: при большем числе автоматизация останавливается
и пишет warning в лог HA. Валидатор предупреждает заранее (`W07`), но не
блокирует — разделить единицу должен человек, проставив второй блок.

## Технические помещения

```python
TECHNICAL_SPACE_TYPES = {"korridor", "special", "recreation"}
```

Проходные пространства, где никто не находится постоянно. Из них
`generate_floor_groups.py` собирает `tex_floor_<n>` — **по умолчанию**;
отключается флагом `--no-tech-groups`.

За рамками: `class` (люди сидят за партами) и `zal` (управляется только
с панелей). Согласовано с владельцем 2026-07-13.

## Areas двух смыслов: помещение и этаж

Решено 2026-07-16. В реестре Areas Home Assistant появляются записи **двух
разных смыслов**, и это надо держать в голове:

| Что | Имя | Кто наполняет |
|---|---|---|
| **Area помещения** | из колонки «Название помещения» | **интеграция**, по устройствам |
| **Area этажа** | `Весь <N> этаж` (`canon.floor_area_name`) | **владелец руками** — кладёт туда групповые светильники этажа |

Зачем нужна вторая: карточка `type: area` на Главной умеет показывать
**только Area**, карточки для Floor в HA не существует. Этаж как таковой —
это Floor, но нарисовать его карточкой нельзя.

**Почему это не конфликт.** Сущность в HA принадлежит ровно одной Area, так что
Area-этажа могла бы конкурировать с комнатными за одни и те же сущности. Не
конкурирует: комнатные наполняет интеграция по **устройствам**, а групповые
светильники этажа (`light.ves_<N>_i_etazh`, `light.tekh_pom_<N>_i_etazh`) —
это YAML-группы **без устройства**, интеграция их никуда не разложит. Попасть
они могут только в Area этажа.

⚠ На этаж приходятся **три** сущности с похожими именами:

```
Floor  «1 этаж»        — запись реестра этажей, группирует Areas
Area   «Весь 1 этаж»   — ves_1_etazh, контейнер для групповых светильников
light  «Весь 1-й этаж» — light.ves_1_i_etazh, сам выключатель
```

`area_id` генерирует HA из имени через slugify — задать его напрямую при
создании нельзя. Поэтому `canon.floor_area_id()` выводит его из того же имени,
что и создаёт: та же схема, что у `floor_light_entity()`, и по той же причине
(один раз уже обожглись, приняв `unique_id` за `entity_id`).

## ⚠ Схема объявлена явно, а не выводится из данных

Машиночитаемый источник правды — **`scripts/_lib/schemas.py`**. Таблицы ниже
описывают то же самое человеческими словами; при расхождении верен код.

`pa.Table.from_pandas()` выводит типы из данных, и если колонка пуста во всей
таблице, ставит туда `null`. На объекте без единой панели `panels_by_group`
получал тип `list<list<null>>` вместо `list<list<string>>`, а `warnings` менял
тип в зависимости от того, были ли предупреждения — то есть **схема файла
зависела от данных**, и генератор сломался бы не у разработчика, а у наладчика
на объекте, который никто не видел.

Поэтому normalize пишет строго по объявленной схеме:

```python
pa.Table.from_pandas(frame, schema=SCHEMAS[name], preserve_index=False)
```

Читающая сторона использует `scripts/_lib/normalized.py` — он сверяет схему
файла с объявленной и внятно ругается на parquet от старой версии.

## `devices.parquet`

| Поле | Тип | Комментарий |
|---|---|---|
| `row_id` | int | номер строки Excel (1-based, с учётом заголовка) |
| `kind` | str | `lamp` \| `sensor` \| `panel` |
| `addr` | str | `X.Y.Z` как в таблице |
| `addr_floor` | Int64 | `X` |
| `addr_bus` | Int64 | `Y` |
| `addr_num` | Int64 | `Z` |
| `entity_id` | str | `light.l_*` \| `sensor.ms_*` \| `event.kp_*` |
| `entity_id_2` | str? | `sensor.il_*` только для `kind = sensor` |
| `group_id` | str | из колонки «Группа» той же строки |
| `space` | str | ffill |
| `room_slug` | str | транслит `space` |
| `floor` | Int64 | колонка «Этаж» |
| `space_type` | str? | нормализованный ключ, `null` если не указан |
| `block` | str? | колонка «Блок»; `null`, если помещение само по себе |
| `dali_bus` | str? | колонка «Шина DALI», справочно |

## `groups.parquet`

`group_id`, `space`, `room_slug`, `floor`, `space_type`,
`zone_light_entity` (= `light.<group_id>`),
`lamps` (list[str] entity_id), `sensors_ms`, `sensors_il`, `panels` (list[str]),
`lamp_count`, `sensor_count`, `panel_count`.

Порядок групп и устройств внутри — как в таблице. Наладчик сверяет отчёты и
YAML со своим Excel, и переупорядочивание сделало бы это невозможным.

## `spaces.parquet`

`space`, `room_slug`, `floor`, `space_type`, `has_valid_type` (bool),
`groups` (list[group_id]), `groups_count`,
`general_light_entity`, `zone_light_entities`,
`sensors_by_group` (list[list[str]]), `panels_by_group` (list[list[str]]),
`sensors_unique`,
`block` (str?), `unit_id` (str), `family` (str?),
`warnings`.

`unit_id` — единица обслуживания: `block`, если задан, иначе `room_slug`.
`family` — `default` \| `hall` \| `special` \| `null` (class и zal не
автоматизируются). См. §6.1.

**`sensors_by_group[i]` относится к `groups[i]`.** Вложенный список пуст, если
у группы нет датчиков — это норма для `zal`, а не пропуск. Именно здесь v1
ломалась: она хранила один датчик на группу и второй молча теряла.

Пример (`105_Актовый зал`):

```
groups           = ["105_1", "105_2"]
sensors_by_group = [[], []]
panels_by_group  = [["event.kp_1_2_1"], ["event.kp_1_2_2"]]
```

## `units.parquet`

Единицы обслуживания — то, на что клонируется набор скриптов (см. §6.1).
Только автоматизируемые помещения (class и zal сюда не попадают).

`unit_id`, `family`, `spaces` (list[str]), `space_type`, `floors` (list[int]),
`sensors_ms` (list[str]), `sensor_count`,
`zone_lights` (list[str]),
`scripts` (list[str] — `script.<unit_id>_<role>`),
`blueprint_on`, `blueprint_off`, `warnings`.

## `normalized_meta.json`

`schema_version = 3`, `generated_at`, `source_file`, `sheet_name`, `stats`,
`columns`, `notes`. В `stats`, помимо устройств и помещений: `units`,
`scripts`, `automations`.

# 8. Что ломается относительно v1

Перечислено для ревью, а не для истории:

1. `is_group_start` — понятия больше нет.
2. «Один датчик = одна группа» — неверно, датчиков может быть несколько.
3. Дедупликация датчиков внутри помещения (v1 молча выкидывала повтор) —
   заменена на **ошибку валидации**: один датчик в двух группах недопустим.
4. Колонка `card_type` — исчезла, заменена на `Тип помещения`.
5. Колонки `(авто)` — исчезли, fallback-логика `_pick_first_existing` больше
   не нужна для них.
6. `generic` и `nearest_variant` fallback — упразднены.
7. `MS_SENSOR_i` в шаблонах карточек индексировался по группам. С 0..N датчиков
   на группу это правило нужно пересматривать вместе с новыми шаблонами.

Старый `data/example.xlsx` остаётся в репозитории как исторический артефакт.
Пайплайн v2 на нём не работает — это ожидаемо.
