# Сделано

## Этап 0–1 — окружение, канон, схема ✅

Dev-окружение на Linux VM, `requirements-dev.txt`, pytest.
`canon.py`: `parse_addr()`, билдеры сущностей, `normalize_space_type()`,

## Этап 2 — `validate_excel.py` ✅

Первый шаг. Ничего не пишет в `normalized/`, печатает отчёт и
`data/validation_report.json`. Коды возврата: `0` — можно генерировать,
`1` — блокирующие ошибки (или предупреждения при `--strict`), `2` — файл не найден.

**Блокирующие ошибки:**

| Код | Правило |
|---|---|
| `E01` | нет листа «Проектная БД» |
| `E02` | нет обязательной колонки |
| `E03` | адрес лампы не в формате `X.Y.Z` (или `None` в колонке «Лампа») |
| `E04` | дубликат адреса лампы |
| `E05` | дубликат адреса датчика |
| `E06` | дубликат адреса панели |
| `E07` | один датчик привязан к двум группам (поверх `E05`) |
| `E08` | группа без ламп |
| `E09` | помещение без ламп / без групп |
| `E10` | в помещении колонка «Датчик» пуста во всех строках (нужен адрес или `None`) |
| `E11` | то же для «Панель» |
| `E12` | устройство в строке без группы |
| `E13` | `group_id` встречается в двух помещениях |
| `E14` | нераспознанное значение в «Датчик»/«Панель» |
| `E15` | блок смешивает разные семейства |
| `E16` | блок задан у помещения, которое не автоматизируется |

**Предупреждения:**

| Код | Правило |
|---|---|
| `W01` | помещение без типа → исключается из карточек |
| `W02` | неизвестный тип помещения |
| `W03` | этаж в адресе ≠ колонка «Этаж» (истина — в адресе) |
| `W04` | шина в адресе ≠ колонка «Шина DALI» |
| `W05` | группа без датчиков (кроме `zal`) |
| `W06` | пустая строка внутри листа |
| `W07` | в единице обслуживания больше 12 датчиков |
| `W08` | автоматизируемое помещение без датчиков |
| `W09` | префикс группы ≠ номер помещения (описка при копировании строки) |

## Этап 3 — `normalize_excel.py` ✅

Длинный формат: **строка = одно устройство**. `device_rows.parquet` упразднён.

**Решение:** normalize отказывается работать на таблице с блокирующими ошибками
(зовёт ту же `validate()`), обход — `--force`. Иначе пайплайн не падает, а тихо
врёт — ровно та проблема, из-за которой затевался переезд.

### 3.1 Жёсткая схема parquet ✅

`pa.Table.from_pandas()` выводил типы из данных: на объекте без панелей
`panels_by_group` становился `list<list<null>>`. Схема файла зависела от объекта,
и генератор сломался бы не у разработчика, а у наладчика. Теперь схема
объявлена явно в `schemas.py`, запись идёт строго по ней.

### 3.2 `show_normalized.py` ✅

Parquet бинарный, глазами не открыть. Режимы: обзор, `--space`, `--group`, `--full`.

## Этап 4 — группы света ✅

Три генератора на новую схему. Фильтры переехали из констант в исходнике в CLI
(`_lib/filters.py`): раньше, чтобы исключить этаж, наладчику пришлось бы править
код, а лаунчер передать их не мог.

**Исправлен дефект v1: висячие сущности.** У `generate_general_groups.py` стояло
`EXCLUDE_SPACE_CONTAINS = ["koridor"]` — артефакт отладки перед ПНР. Коридоры не
получали общую группу, но `generate_floor_groups.py` всё равно тянул их
`general_light_entity` в группу этажа: в HA появлялась группа со ссылкой на
несуществующую сущность.

`TECHNICAL_SPACE_TYPES = {korridor, special, recreation}` — проходные
пространства. `tex_floor_<n>` создаются по умолчанию, отключаются
`--no-tech-groups`.

## Этап 5 — Areas и Floors ✅

Основано на `Auto-area-HA`, который уже читал наш `spaces.parquet`. Разрезан
надвое: `transform.py` → наш генератор, `ha_client.py` → транспорт деплоя.

**Выброшен `TRANSLATION_DICT`:** он восстанавливал русское имя из транслита по
словарю (`103_vestibiul` → `103_Vestibiul`, латиница). Костыль под таблицу v1,
где `space` выглядел как `403_кабинет_медиц`. В v3 колонка «Название помещения»
уже человеческая — имя Area берём прямо из неё, транслит уходит в `aliases`.

Шаг офлайновый: тест `test_generator_makes_no_network_calls` это стережёт.

## Этап 6 — единицы обслуживания ✅

Основание: `uncledrew/logic-contol-system/` — схема
`Датчик → Blueprint → Zone Manager → Скрипт`. Blueprint'ы и скрипты статичны;
генерировать надо клоны скриптов и автоматизации под них.

**Колонка «Блок».** Единица обслуживания = помещение (если блок пуст) либо все
помещения одного блока. Правила заказчика («лестницы по стояку», «санузлы с
ближайшими») опираются на пространственную информацию, которой в таблице нет:
эвристика работала бы на одном объекте и молча врала на другом. Решение
принимает проектировщик, глядя на план.

Семейства: `korridor`/`recreation` → `default`, `hall` → `hall`,
`special` → `special`, `class`/`zal` → не автоматизируются.

## Этап 7 — скрипты ✅

`generate_scripts.py` клонирует шаблоны из `templates/scripts/` по единицам.

**Зачем клонировать:** один экземпляр скрипта в HA — это одна очередь. При
тысяче датчиков вызовы копятся, и свет отстаёт от человека; `mode: queued` не
спасает. У каждой единицы свой набор — они друг друга не ждут.

Меняются ровно две вещи: корневой ключ (из него HA делает entity_id) и `alias`
(иначе в UI было бы двести одинаковых «Шаблон скрипта»). Тело не трогаем —
имён в нём нет, конфигурацию скрипт получает параметрами.

## Этап 8 — автоматизации ✅

`generate_automations.py` — экземпляры blueprint'ов, по две на единицу (ON и OFF).
Плюс копии blueprint'ов в `data/blueprints/` для деплоя.

**Пакет, а не `automations.yaml`:** туда HA пишет автоматизации, созданные через
UI, и перезапись стёрла бы ручную работу наладчика.

**У `special` OFF вход один** (`off_script`), у `default` и `hall` — два.
Имена входов описаны явно в `BLUEPRINT_INPUTS_BY_FAMILY` и сверяются тестом с
самими blueprint'ами: опечатка не уронила бы генерацию — HA молча
проигнорировал бы вход, и автоматизация осталась бы без скрипта.

# Карточки Lovelace

## Карточки Lovelace — сделано (2026-07-15)

`generate_lovelace_cards.py` (v3): у каждого `space_type` **своя раскладка**
(согласовано с владельцем). Собирается методом «обёртка + блоки»: обёртка типа
(`templates/lovelace/<type>/wrapper.yaml`) с маркерами, атомарные плитки
(`_blocks/`), генератор разворачивает повторяющиеся части по правилу типа.

| Тип | Раскладка | Источник |
|---|---|---|
| korridor | пары `свет\|датчик`, балансная нарезка по 3 зоны, макс. 2 сетки в ряд с переносом | данные |
| hall | = korridor | данные |
| special | вертикальный список пар `свет\|датчик` | данные |
| class | грид N колонок: ряд света / ряд датчиков / ряд подписей (заглушка `—`) | данные + подпись руками |
| zal | группы света + **захардкоженные** пресеты-сценарии | данные + хардкод, mushroom/card_mod |
| recreation | общий свет + узкие группы (mushroom) + датчики в 2 колонки | данные, mushroom |

- «датчиков ≠ зон» решено структурно: 0 → пустая ячейка, >1 → стопкой.
- Помещение без валидного типа карточку не получает (`has_valid_type`, W01).
- Кнопка в лаунчере есть, **в Build All не входит**.
- Требование объекта: для zal/recreation на HA должны стоять HACS-карты
  mushroom + card_mod.
- Тесты: `tests/test_generate_lovelace_cards.py` (нарезка, перенос 7–9 зон,
  сборка всех типов, валидность YAML).

### Views дашборда и деплой карточек (2026-07-16)

Экспорт в `.txt` с копипастом руками (наследие v1) **упразднён**. Карточки
раскладываются по views дашборда и деплоятся, как остальные артефакты.

| Что | Куда |
|---|---|
| view на этаж (`zm-floor-<N>`) | компактные карточки помещений этажа |
| subview на пространство (`zm-space-<room_slug>`) | полная карточка, `subview: true` |

- **Компактная карточка** (`_blocks/compact_card.yaml`) — заголовок + общий свет
  + кнопка «Подробнее» с `tap_action: navigate` в subview. Без неё subview был бы
  недостижим: он скрыт из вкладок по определению.
- Оба вида views — `type: sections`: наши карточки используют `grid_options`,
  вне sections-view они не разложатся.
- **Выход:** файл на view в `data/lovelace/` (`--dashboard` задаёт url_path,
  из него строятся navigate-пути; на каждом объекте свой).
- **Деплой** — цель `lovelace`, канал **WebSocket** (не файлы в `/config`):
  `lovelace/config` (прочитать) → `ha_views.merge_views` → `lovelace/config/save`.
  Дашборд остаётся **storage/UI-режимным**, редактируется руками как прежде.
- **Слияние:** свои views опознаются по префиксу пути `zm-` (аналог `zm_` у
  файлов). Чужие views (Энергомониторинг, Ошибки) не трогаются; наши встают на
  фиксированную позицию — **в начало** (`INSERT_AT = 0`), чтобы регенерация не
  уносила этажи в хвост. Идемпотентно.

  ⚠ Было `INSERT_AT = 1` («сразу после первого view»), пока Главная
  принадлежала владельцу. С тех пор как Главную генерируем мы, первой обязана
  быть она: дашборд открывается на первом view, и туда же ведёт кнопка «назад»
  с этажей.
- **Рестарт не нужен** — конфиг дашборда применяется сразу.
- ⚠ `lovelace/config/save` требует токен **администратора** и перезаписывает
  дашборд целиком — поэтому логика слияния вынесена в чистый
  `_lib/ha_views.py` и покрыта `tests/test_ha_views.py` (идемпотентность,
  сохранность чужих views, порядок, сводка для dry-run).

# Оформление лаунчера

## Оформление лаунчера ✅ (2026-07-17)

Светлая база, пурпурный акцент, приборная геометрия — `launcher/ui/theme.py`,
один `apply_theme(app)` на всё приложение.

Жанр на светлом фоне держится **формой, а не цветом**: неон — это контраст
яркого пятна с темнотой, а на светлом акцент обязан быть тёмным, иначе его не
прочесть (неоновый циан `#22d3ee` на белом даёт 1.7:1). Работают: нулевое
скругление, тонкие рамки, циановая рейка и полоса у секций, заголовки
заглавными с разрядкой, моноширинный лог, скан-линии подложки.

### Два акцента, и у каждого своя работа

| Цвет | Роль | Где |
|---|---|---|
| `CYAN` `#0e7490` | **структура** | заголовки секций, рейка сверху, полоса слева, рамка подсказки |
| `ACCENT` `#be185d` | **действие** | наведение, нажатие, фокус, отмеченный флажок, выделение |

Свойство, ради которого это затевалось: **в покое окно холодное, пурпур
вспыхивает ровно там, где вы трогаете.** Смешаете роли — оба цвета станут шумом.
Стережёт `test_action_colour_is_used_only_for_action`: он разбирает QSS и
требует, чтобы у каждого правила с `ACCENT` в селекторе было интерактивное
состояние. Первым же запуском поймал подсказку с пурпурной рамкой.

⚠ **Семантика важнее акцентов.** Ошибка/успех/предупреждение инженер читает
периферийным зрением. Пурпур и красный `ERROR` оба красноватые, поэтому ещё и
разведены по местам: акцент — только хром, красный — только лог и статусы.

### Кнопки: номера в колонку

`[01] VALIDATE EXCEL`, выравнивание **влево**, без `letter-spacing`.
При центрировании подпись каждой кнопки встаёт по своей ширине, и номера не
собираются в колонку — пляшут по горизонтали и выглядят оторванными от текста.
Префикс у всех ровно 4 символа (`[01]`, `[ +]`, `[ ▶]`, `[ ↑]`, `[ ·]`) — только
так моноширинный шрифт даёт ровный столбец.

Номера — только у шагов пайплайна: `LOVELACE CARDS` их не получает, иначе
соврал бы про порядок (он в Build All не входит).

⚠ Номер отдельным цветом внутри кнопки **невозможен**: `QPushButton` не
рендерит рич-текст, показывает разметку как текст (в отличие от `QLabel`).
Понадобился бы свой `paintEvent`.

### Капканы Qt, на которых уже наступили

| Что | Чем кончается |
|---|---|
| QSS **файлом `.qss`** | нужен `--add-data`; забыли — EXE без стилей, а у разработчика всё работает. Поэтому QSS — питоновской строкой |
| Свой шрифт (Orbitron и т.п.) | лицензия + `--add-data` + тот же капкан. Только системные: Consolas → Courier New |
| `text-transform` в `QGroupBox::title` | к `QLabel` применяется, до субконтрола НЕ доходит. Заголовки заглавные в самих строках (`.upper()`) |
| Стилизовать `QSpinBox::up-button` | Qt перестаёт рисовать нативную стрелку, а своя требует `image`. CSS-трюк «треугольник из рамок» в Qt не работает. У единственного QSpinBox стрелки выключены через `setButtonSymbols(NoButtons)` |
| Покрасить `QPushButton` | нативная отрисовка отключается ЦЕЛИКОМ: `:hover`/`:pressed`/`:disabled` описываем руками. `:disabled` не косметика — лаунчер гасит кнопки на время работы пайплайна |
| Тронуть рамку `QGroupBox` | обязан стилизовать и `::title`, иначе заголовок врастает в рамку |
| Полосы через `qlineargradient(spread:repeat)` | координаты **относительные** (0..1 от размера виджета): волоски в маленьком окне, жирные ленты в развёрнутом. Проверено рендером на двух высотах. Скан-линии мостятся плиткой `QPixmap` в пикселях, через `QPalette.Window` — в QSS нет способа замостить фон картинкой из памяти, только `url(файл)` |
| Рич-текст в `QPushButton` | не рендерится, разметка видна как текст. Только `QLabel` умеет |
| `setDevicePixelRatio` + отрисовка в `rect * ratio` | масштаб применяется **дважды**: рисунок раздувается и обрезается по краю холста. Тем сильнее, чем крупнее масштаб экрана — на 100% не видно вовсе. Холст заводим в физических пикселях, а РИСУЕМ в логический прямоугольник |
| Подкласс `QWidget` + свой `setStyleSheet` | **не рисует его вовсе** — ни фона, ни рамки. Готовые виджеты делают это внутри, голый `QWidget` обязан прогнать `QStyle.PE_Widget` в `paintEvent` руками. Правило при этом выглядит рабочим |
| Срезанные углы 45° (главный маркер жанра) | QSS умеет только скругление. Нужен свой `paintEvent` на каждый тип виджета — отклонено владельцем |

### Графика: шапка, иконка, уголки-скобки

`launcher/ui/decals.py` — SVG **питоновской строкой**, `launcher/ui/widgets.py` —
`HeaderBar` и `BracketGroupBox`.

- **Шапка**: имя, версия (`launcher/__version__`), декаль-схема этажа справа.
  Версия на экране не для красоты: наладчик звонит с объекта, и первый вопрос —
  какая у него сборка. Суффикс `-dev` держится до тега `v3.0.0`.

  > **Исход (2026-07-17):** суффикс снят перед мержем, `__version__ = "3.0.0"`.
  > Дальше версию правим руками перед каждым тегом, а CI сверяет её с тегом и
  > падает при расхождении. Сам тег на момент записи ещё не поставлен.
- **Иконка окна**: шина DALI с тремя узлами, растры 16–128px (Qt берёт ближайший
  и масштабирует — с одним 64px иконка в заголовке мылила бы). Раньше в панели
  задач стояла заглушка Qt.
- **Уголки-скобки**: только снизу. Сверху уже рейка и заголовок — скобка налезла
  бы на них. Рисуются ПОВЕРХ нативной отрисовки, поэтому весь QSS секции
  продолжает работать.

**Декаль** — цепочка иконок [Lucide](https://lucide.dev) (ISC). Выбрана
владельцем из четырёх вариантов, они остались в `docs/design/decals/` вместе с
рендерами в контексте шапки. Отклонены: схема этажа (не читалась планом),
спектр, осциллограмма, радар.

⚠ **ISC — не «без обязательств».** Разрешает коммерческое использование даром и
без спроса, но требует копирайт и текст лицензии **во всех копиях**. Лаунчер
уезжает одним EXE, поэтому текст лежит СТРОКОЙ в `decals.LUCIDE_NOTICE`: файл
рядом потребовал бы `--add-data`, и забытый флаг превратил бы нарушение лицензии

⚠ **Откуда графику не берём:** Pinterest (доска чужих картинок без автора и

право. CC0-фильтр там есть, но за ним пиксель-арт для игр — растр и чужая
эстетика. И главное: у нас **коммерческое использование** — инструментом
настраивают освещение за деньги, независимо от того, продаётся ли он сам.

### Как это проверяется

QSS молчит: правило с опечаткой не падает и не логируется, просто не работает.
`tests/test_theme.py` ловит машинно — Qt ругается «Unknown property» в свой
message handler, тест его слушает.

⚠ Разбор QSS **ленивый и по типам виджетов**: не создашь `QTextEdit` — правило
`QTextEdit` не разберётся, и опечатка в нём пройдёт мимо. Поэтому список
виджетов выводится из самого QSS, а не перечисляется руками.

Тест требует PySide6 и пропускается, если он не импортируется (на Linux-VM нет
`libEGL`); на машине наладчика, где лаунчер и живёт, отрабатывает.

## Главная страница ✅ (2026-07-17)

Решено 2026-07-16. Источник требований — Главная с объекта «Колледж Химки»
(4 этажа, ~76 помещений), которую владелец дал **как образец желаемого**.

⚠ Это **не объект миграции**. Делаем для песочницы и будущих объектов, где
ничего ещё не создано: ни помощников, ни ручных views, ни Areas-этажей.
Поэтому совместимость с химкинскими именами не требуется — конвенции
выбираем свободно, «с чистого листа».

### Что убираем из текущей Главной

- кастомный фон (`background:`);
- карточку «Выбор режима работы колледжа» целиком (`input_select.mode_work_colhim`,
  `input_button.but_save_mode`, `sensor.tekushchii_rezhi_raboty_kolledzha`,
  `switch.dali_auto_brightness_master`);
- виджеты шапки (`badges`: режим, `input_text`, календари).

### Блок на этаж

1. `type: area` с картинкой и переходом на этажный view;
2. `horizontal-stack`: свет этажа + свет тех.помещений;
3. список выбора помещения (`input_select`);
4. **кнопка перехода** на выбранное помещение.

### Навигация: 76 `conditional` → 1 markdown-карточка на этаж

Было: `input_select` + по одной `conditional`-карточке на КАЖДОЕ помещение
(39/13/12/12 = 76 штук), каждая с зашитым именем и путём. Причина не в дурном
вкусе: **`navigation_path` у штатных карточек — статическая строка**, шаблоны
в неё не подставляются, и 76 условий были единственным штатным способом
выбрать нужную строку. Настоящий изъян — ручная синхронизация: опции
помощника, условия и пути жили в трёх местах и обязаны были совпадать
символ в символ.

Решение — **markdown-карточка**: она поддерживает Jinja и ссылки на внутренние
дашборды. Одна карточка на этаж вместо 39, карту «имя → слаг» генерируем мы.
Проверено на живом HA — переход работает. Вид утверждён владельцем: ссылка
`display: block` (кликается вся карточка), текст слева, `›` справа через
`a::after`, оформление под этажную кнопку.

⚠ Стилизуется только словарной формой card_mod с `$`: ссылка лежит внутри
`ha-markdown`, у которого свой shadow-root, и строковый `style: |` до неё
не достаёт.

Рассмотренные и отклонённые:
- `config-template-card` — шаблонит любое поле, 1 карточка на этаж, но
  **несущая** HACS-зависимость (стабильный релиз 2021, работа в бете);
- `browser_mod` — то же, несущая зависимость;
- список `weblink`-строк — нативно и в один клик, но это простыня: владельцу
  важно, чтобы **видна была одна выбранная кнопка**, а не весь список;
- генерировать те же 76 `conditional` — рассинхрон уходит, но конфиг растёт
  как O(помещений): ~480 на 12-этажном объекте.

### Сервисные страницы: посев, а не генерация ✅ (2026-07-17)

Секция «Сервис» под этажами: две кнопки — расписание и конфигурация
(`canon.SERVICE_VIEWS`, вид в `_blocks/main_service_block.yaml`). Штучные, из
таблицы не выводятся, — как пресеты зала.

⚠ **Пути заготовок НАМЕРЕННО без префикса `zm-`.** Наполнять эти страницы будет
владелец, значит они его. Дай им `zm-` — и `merge_views` выкинул бы их вместе с
нашими, а на место наполненной страницы легла бы пустая заготовка.

Поэтому деплой их не генерирует, а **высевает**: нет такого пути на дашборде —
создаём пустую заготовку, есть — не трогаем никогда. Правило слияния остаётся
однострочным (`zm-` наше, остальное чужое), без единого исключения. Стережёт
`test_service_stubs_are_not_ours`.

Тип раздела заготовки — в `canon.SERVICE_VIEWS.view_type`:

| Страница | Тип | Каркас |
|---|---|---|
| `raspisanie` | `panel` — «Панель (1 карточка)» | `cards: []` |
| `konfiguratsiia` | `sections` | `sections: [{type: grid, cards: []}]` |

У панели карточки лежат прямо в `cards`, секций у неё нет вовсе — каркас обязан
соответствовать типу, иначе HA отрисует раздел пустым без всяких подсказок.
Тип задаём мы, а не владелец: страницу высевает деплой, и сменить тип он сможет
только руками через UI — наполненную страницу мы больше не трогаем.

Отклонено из образца с объекта:

| Что | Почему |
|---|---|
| `type: picture` с `media-source://.../local/*.jpg` | бинарник, которого нет ни в репо, ни на чистом объекте → битая картинка. Заменено кнопкой с иконкой |
| `metadata` / `navigateIds` в `image` | мусор визуального редактора: `hui-picture-card` его не читает, `image` принимает media-source строкой |
| `grid_options` внутри `vertical-stack` | инертны: их получают только прямые дети секции |
| `theme: Nordic Blue Light` | несущая зависимость от темы на объекте |
| путь `/lovelace/...` | это дефолтный дашборд, а не наш. Пути строим от `--dashboard` |

Ширина: секция «Сервис» — `column_span: 1` (12 колонок), у блоков `columns: 6`
(два в ряд). У секции этажей `column_span: 2`, то есть 24 колонки: число в
`grid_options` считается от ширины СВОЕЙ секции.

**Выравнивание по высоте — убрали причину, а не последствие.** При `rows: auto`
высота блока считается по содержимому (число в `rows` включает карточке класс
`fit-rows` и жёсткую высоту, `auto` этот расчёт не запускает). «Настройка
конфигурации» не влезла в колонку, перенеслась на вторую строку — и ряд поехал.

Вместо подгонки высот (жёсткий `rows`, card_mod с пикселями) убрали то, что
переносится: подписи у кнопки нет вовсе (`show_name: false` — она дублировала
заголовок над собой), заголовки короткие, в одно слово. Высоты совпадают по
построению, без CSS и магических чисел. Стережёт
`test_service_blocks_have_nothing_that_can_wrap`.

  с группами света. Новых механик не нужно.
- **Имя из ключа, а не из `name`.** В YAML `entity_id` берётся из ключа
  (`vacant_delay:` → `input_number.vacant_delay`), поэтому предсказуем. Через UI
  он выводится из отображаемого имени — та же ловушка, на которой мы уже
  обожглись с группами этажа. Ещё один довод за YAML-пакет.
- ⚠ **Грабля, если помощник уже существует:** объявим в YAML одноимённый с
  заведённым через UI — HA не сольёт их, а создаст второй объект
  (`input_number.vacant_delay_2`). На чистом объекте вопрос не стоит, но на
  объекте с ручными помощниками их надо удалить до первого деплоя.
- Значения по умолчанию (`vacant_delay` — сколько секунд?) взять неоткуда:
  либо константа в каноне, либо колонка в таблице.
- `regim_auto_<N>` и `input_select` навигации требуют списка этажей — он у нас
  есть; остальные штучные.

# Фильтр навигации по типу помещения ✅ (2026-07-20)

Задача владельца: на объекте в 76 помещений список на этаже длинный, найти
нужное трудно. Требовалось выбирать по типу, показывать только выбранное и
уметь выбрать несколько типов сразу.

## Помощники, а не своя JS-карта

Свою карту рассматривали всерьёз — у владельца уже есть опыт vanilla-карт в
другом проекте, и доставка оказалась не барьером: файл кладётся по SFTP в
`/config/www/`, ресурс регистрируется по WebSocket командой
`lovelace/resources/create` (поля `url`, `res_type`; проверено по
`lovelace/const.py`). Оба механизма у нас уже есть, админ-токен тоже.

Отклонили, потому что задача этого не требует: «несколько типов сразу» =
шесть `input_boolean`, нативно и тестируемо. Карта решала бы задачу, которой не
ставили.

| | Помощники | Своя карта |
|---|---|---|
| Новый код | ~40 строк генератора | ~200 JS + ~100 деплоя |
| Тестируется нашим pytest | да | **нет**, только глазами |
| Кто чинит | правим YAML | только мы |
| Сущностей в реестре | +7 | 0 |

⚠ **Один настоящий довод за карту всё же есть, и он не про экономию строк:**
фильтр — это состояние ВИДА, а не объекта. В `input_boolean` он общий:
отфильтровал наладчик на планшете — оператор видит то же. Карта держала бы его
локально в браузере, что семантически правильно. Решили сначала пожить с
общим и узнать из практики, мешает ли это. Мешает — карта пишется потом, ничего
не переделывая: она заменит фильтр в тех же местах.

Мультивыбора в HA нет вовсе: `input_select` однозначен по определению, а HACS
`input_multiselect` требует и бэкенд-интеграцию, и свою карточку — несущая
зависимость того же класса, что уже отклонённые `config-template-card` и
`browser_mod`.

## Помощники общие на объект, а не на этаж

Обход «только коридоры» не должен сбрасываться при переходе между этажами.
Заодно 7 сущностей вместо 7×N: на четырёхэтажном объекте это 7 против 28.

## «Все» — отдельный помощник с `initial: on`

Не «ничего не отмечено = показать всё». Две причины.

**Первая — правило проекта: объект создаём, логику нет.** Автоматизаций за
помощниками мы не пишем, поэтому «Все» обязан работать через одни лишь условия
видимости. Вариант «все выключены» выражается как `or` из шести `and`-условий
НА КАЖДОЙ карточке — семь условий против двух, помноженные на число помещений.

**Вторая — `initial`.** Без него после первого старта HA все фильтры выключены,
ни одна карточка условие не проходит, и этажные страницы оказываются ПУСТЫМИ.
Выглядит как поломка генератора, а не как «фильтр не настроен».

Поведение прощающее: «Все» включён — видно всё, что бы ни было отмечено ещё.
Гасить остальные он не умеет и не должен.

## Два механизма, по месту

| Где | Как | Почему так |
|---|---|---|
| Этажные страницы | `visibility` у карточек | Они `grid` и прямые дети секции — условие работает, а секция схлопывает скрытые без дыр (`.card:has(> *[hidden])`) |
| Главная | не трогаем | см. ниже |

⚠ `visibility` работает ТОЛЬКО потому, что карточка помещения — прямой ребёнок
секции. Внутри стеков и гридов свойство игнорируется. Завернёте карточки во
что-нибудь ещё — фильтр молча перестанет работать, и выглядеть это будет как
«кнопки не нажимаются».

## Главную не трогаем (вариант C)

Рассматривали панель + список помещений по типу со ссылками. Отклонено
владельцем: помощники общие, значит фильтр, выставленный на любой этажной
странице, уже действует везде. Главная отвечает на вопрос «на какой этаж
пойти», а не «какие тут коридоры»; панель добавила бы ~200px самой посещаемой
странице ради сценария, который на этажной уже закрыт.

## Временное: второй фильтр списком

По просьбе владельца рядом с плитками живёт `input_select.nav_type_pick` — тот
же набор типов одним выпадающим списком. Причина: плитки занимают ~200px,
список ~56px, и владелец хочет сравнить оба на живом объекте.

⚠ Список ОДНОЗНАЧЕН: «Коридоры + Залы» одновременно он не умеет. Это и есть
предмет сравнения — стоит ли мультивыбор своего места на экране.

Как два фильтра уживаются: условия видимости верхнего уровня складываются по И,
поэтому карточка видна, только если её пропустили ОБА. Каждый глушится своим
«Все», и это даёт режим сравнения:

```
булевы на «Все»  ->  рулит список
список на «Все»  ->  рулят плитки
оба сужены       ->  пересечение
```

⚠ **Убирая проигравший вариант, снимите и его условие видимости** в
`build_compact_card`. Уберёте только помощника — карточки останутся привязаны к
несуществующей сущности, и этаж окажется пустым. Пометки «ВРЕМЕННОЕ» стоят в
четырёх местах: канон, `generate_helpers`, `nav_filter.yaml`,
`build_compact_card`.

## Как это проверяется

`tests/test_nav_filter.py` — 15 тестов. Ядро: вычислитель `visibility` по
правилам HA (условия верхнего уровня по И, внутри `or` по ИЛИ, `state` может
быть списком — сверено с `validate-condition.ts`). Без него проверять фильтр
пришлось бы глазами на живом объекте, а каждая проверка — деплой с рестартом.

Фильтр держится на сцепке трёх вещей, и все три разъезжаются молча: помощник в
пакете, плитка в панели, условие у карточки. Поэтому почти каждая проверка
сверяется с ДРУГИМ концом сцепки — с фактическим выходом соседнего генератора
или с `spaces.parquet`. Сверка с каноном была бы тавтологией: он подтвердит имя
и у сущности, которую никто не создаёт.

| Поддержано в HA | не заявлено | штатный сценарий |

Я рекомендовал плоский. Владелец выбрал **вложенный**, приняв допущения явно —

Про поддержку вложенности: документация HA о ней молчит вовсе. Проверено по

состояние дочерней через `hass.states.get()`, защиты от рекурсии и оптимизаций
нет. Работать будет, но это не заявленный сценарий.

## Три допущения, записанные в код

Все три таковы, что через полгода их найдут как баг, поэтому лежат в каноне
рядом с функцией, а не только здесь.

**Яркость — среднее от средних по этажам.** `reduce_attribute` берёт значения
участников, а у этажной группы это уже усреднённая величина. При разном числе
помещений на этажах цифра смещена. Вкл/выкл точен, а на Главной группа выведена
бейджем, где ползунка нет, — поэтому цена невелика. Понадобится честная
яркость: состав придётся сделать плоским.

**Техгруппы в состав не входят.** Они подмножество этажных, и техпомещение
попало бы в объект дважды. Ошибка тихая: сумма светится, состав глазами не
виден. Стережёт `test_tech_groups_are_not_included`.

**Помещение без этажа выпадает.** Оно не попало ни в одну этажную группу.
Случай виден в отчёте шага — генератор про такие помещения уже печатает
предупреждение.

## Имя и entity_id

«Весь объект» → `light.ves_obekt` через slugify **от имени**, `unique_id:
object_all`. Не наоборот: `light.object_all` не существует — это тот самый
капкан, на котором бейдж этажа уже висел в пустоте (см. «`unique_id` — это не
`entity_id`» в `data_model.md`).

Название выбрал владелец из двух вариантов. «Весь объект» вместо «Всё здание»,
хотя слово «объект» в проекте уже занято другим смыслом — так называется здание
целиком в доках и в поле лаунчера. Совпадение здесь скорее помогает: оператор и
наладчик зовут его одинаково.

## На одноэтажном объекте группа дублирует этажную

И всё равно создаётся. Бейдж на Главной захардкожен: без сущности он покажет
«недоступна». Дубль безвреден, отсутствие заметно.

## Бейдж, а не карточка

Решение владельца. Компактно, и ползунок яркости здесь всё равно был бы

Попутно нашлось: `badges_position` и `badges_wrap` в шапке Главной стояли

# Оркестратор здания (Building Automation) ✅ (2026-07-22)

Появляется отдельная кастомная интеграция — **Оркестратор здания**. Она берёт
слой, которого у нас нет: режимы работы здания (Урок / Перемена / Не рабочее
время / Окошко), их расписание и разрешение либо запрет работы датчиков. Мы —
владелец всего, что выводится из таблицы (Floors, Areas, группы, скрипты,
автоматизации, blueprint'ы); Оркестратор — владелец того, что вычисляется в
рантайме.

Полный текст договорённостей — три документа рядом:
`contract-ha-lighting-compilers.md` (требования от автора Оркестратора),
`contract-reply-01-gate.md` (наш разбор) и `contract-answer-01-gate.md` (его
ответ и финальные решения). Ниже — почему сделано так, а не иначе.

## Гейт: fail-open только шаблоном, не `condition: state`

Автоматизациям датчиков добавлен вход `ba_gate_entity` — `binary_sensor`
Оркестратора, разрешающий работу на этаже. Условие стоит первым, до любых
действий, во **всех шести** blueprint'ах и в **обеих** ролях (ON и OFF).

Ключевое решение — форма условия:

```yaml
variables:
  ba_gate_entity: !input ba_gate_entity
condition:
  - condition: template
    value_template: "{{ states(ba_gate_entity) != 'off' }}"
```

Контракт (§3.1) предписывал `condition: state` + `condition: not`, считая, что
на отсутствующей сущности это даёт fail-open. **Проверка по исходникам HA
показала обратное.** `condition: state` на несуществующей сущности не
возвращает `False`, а бросает `ConditionErrorMessage`; `not` её не глотает, а
перебрасывает; автоматизация ловит `HomeAssistantError` и **не выполняется**.
То есть в момент выкатки, когда Оркестратора ещё нет ни на одном этаже, весь
свет по датчикам встал бы. Разбор со ссылками на строки — в
`contract-answer-01-gate.md`; автор подтвердил его по целевой версии 2026.5.4 и
снял запрет на `condition: template`.

Почему именно `states(...) != 'off'`, а не что-то «аккуратнее»:

- `states()` для отсутствующей сущности возвращает `'unknown'` и **не бросает** —
  fail-open держится во всех трёх случаях: нет сущности, `unavailable`,
  `unknown`. Это ровно то, что нужно: до установки Оркестратора и при каждой
  его перезагрузке датчики работают как сейчас.
- `has_value(...) and ...` — **ловушка**: `has_value()` даёт `False` и на
  `unavailable`/`unknown`, вернув нас в fail-closed. Тест
  `test_blueprint_gate_condition_is_failopen_template` отвергает и его, и
  `== 'on'`.
- `!input` внутри `value_template` **не разворачивается** (подстановка идёт по
  узлам YAML, строка шаблона нетронута) — проброс через `variables` обязателен,
  и они рендерятся раньше условий.

Цена, которую надо знать: опечатка в имени гейта не заметна — `states()` вернёт
`'unknown'`, условие всегда пройдёт, Оркестратор никогда не сможет запретить
датчики, и в логе тихо. Миримся с этим сознательно: имя подставляет генератор
по формуле `canon.ba_gate_entity()`, руками его не вводят, так что опечатке
взяться неоткуда. Тихая деградация до текущего поведения несравнимо мягче
темноты во всём здании.

**Гейт в обеих ролях — не перестраховка.** Без него в OFF-блюпринте режим, где
датчики запрещены и свет статически горит на 100 %, всё равно погас бы по
таймауту вакансии. Стережёт `test_gate_present_in_both_roles`.

**Многоэтажная единица.** Единица обслуживания (блок через лестничный стояк)
может жить на нескольких этажах, а гейт один на этаж. Берём минимальный
(`min(floors)`) и **громко предупреждаем** в отчёте шага — наладчик должен
проверить, тот ли этаж. Молчаливый выбор нашли бы нескоро.

## Одна световая сущность в Area

Генератор теперь сам назначает свет в Areas и метит их — раньше свет в Area
этажа клал владелец руками, а комнатные наполняла только интеграция по
устройствам. Политика: **в каждой Area ровно одна сущность домена `light`** —
общая группа помещения в комнатной Area, группа этажа в агрегатной.

Зачем инвариант: сервисный вызов с целью `area_id` разворачивается во **все**
световые сущности Area. Лежи там и лампы, и группа — каждая лампа получила бы
две команды, прямую и через группу, удвоив трафик на шине DALI, который весь
проект и экономит. Поэтому в Areas не попадают ни отдельные лампы, ни зонные
группы, ни объектная группа.

Инвариант при деплое **не проверяем** — за ним следит Оркестратор в рантайме
(решение владельца). Наша сторона лишь не создаёт нарушений: свет строится
билдерами канона (`general_light_entity`, `floor_light_entity`), по одному
полю `light` на Area.

## Метки: этажная и по типу

Оркестратор читает реестр, а не наши файлы, поэтому размечаем его метками:

- `ba_floor_area` — на агрегатных Areas этажей. Отличает контейнер этажа от
  помещения. **Опознавать по имени нельзя:** `area_id` выводится из имени при
  создании, а переименование в интерфейсе HA его не меняет — связь по имени
  разъехалась бы. Без метки Оркестратор отправил бы команду и на этажную
  группу, и на каждое помещение этажа — двойное воздействие с другой стороны.
- `ba_type_<тип>` — на комнатных Areas, по колонке «Тип помещения». Позволяет
  задать профиль режима сразу всем помещениям одного типа («во всех коридорах
  на Перемене свет 100 %») вместо ручной настройки каждого.

Имена меток **ASCII не по стилю, а по необходимости:** `label_id` HA выводит из
имени через slugify, от кириллицы он был бы непредсказуем, а имена жёстко зашиты
в Оркестраторе.

**Тех.помещения остаются без Area** (решение владельца): в Area этажа уже лежит
группа этажа, а второй свет сломал бы инвариант. Отдельную Area под них не
заводим — управление ими по расписанию не требуется. Метку opt-out, которую
автор было запросил, он же и отозвал: раз своей Area у тех.помещений нет,
Оркестратор их не видит и метить нечего. Стережёт
`test_tech_group_is_never_assigned`.

## Два прохода деплоя — это норма, а не сбой

Назначить свет можно только той сущности, что уже есть в реестре HA. На **первом**
деплое групп света там нет: пакеты только легли на диск по SFTP, а Home Assistant
деплой **не перезапускает** (решение владельца). Значит первый проход создаёт
Areas и метки, но часть назначений отложит, честно перечислив отсутствующие
сущности и подсказав перезапустить HA. Второй проход, после рестарта, их доназначит.

Это заложено явно: отсутствующая сущность — **не ошибка**. `entities_missing` в
плане печатается до выхода по идемпотентности, иначе на повторном деплое
сообщение бы потерялось.

## Идемпотентность разметки — два тонких места

- **Метки дописываются, а не заменяют.** `config/area_registry/update` подменяет
  список меток целиком. Не подмешивай мы уже стоящие — деплой снёс бы всё, что
  владелец навесил в интерфейсе HA, и заметил бы он это нескоро.
- **`is_empty` учитывает разметку.** Иначе повторный деплой, где все Areas уже
  созданы, счёл бы работу выполненной и молча пропустил назначения — ровно тот
  второй проход, ради которого его и запускают.

Дифф — чистая логика без сети (`build_areas_plan`), тестируется отдельно от
клиента, как и создание Areas.

## Правка 5 отложена: у `regim_auto` есть второй потребитель

Контракт предлагал прекратить генерацию `input_boolean.regim_auto_<N>` — их
логику раньше дописывали руками четырьмя автоматизациями, и гейт их заменяет.
Но у них есть потребитель, которого контракт не видел: **бейдж режима на каждом
этажном view** (`floor/view.yaml`). Просто перестать генерировать — оставить
дырку на каждом этаже.

Отложено до появления Оркестратора. Тогда же в бейдже сущность меняется — и
**не на гейт**: бейдж кликабельный, а гейт вычисляемый и read-only. Замена —
`switch.building_automation_floor_<N>_automatic` (тумблер, ON = автоматика).
Полярность совпадает с legacy `regim_auto`, диспетчер не переучивается.

⚠ Удаление из генерируемого файла **не удаляет** уже созданные на объекте
сущности — на существующих объектах понадобится ручная уборка. На новых
вопрос не возникает.

## Что осталось непроверенным

Сетевые команды (`label_registry`, `entity_registry/update`, `area_registry/update`)
и гейт проверены только на моке и на уровне текста YAML — поведение живого HA
подтвердит первый деплой на объекте. Риск по гейту ниже: конструкция согласована
с автором и сверена им по исходникам целевой версии.
