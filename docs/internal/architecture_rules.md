# Внутренние правила генераторов проекта освещения

⚠️ Документ предназначен для разработчиков проекта.
Не является пользовательской документацией.

Здесь зафиксированы правила и допущения, принятые при разработке
скриптов генерации YAML и Lovelace-карточек.

📌 1) Источники данных и их смысл

Excel (.xlsx) — единственный источник “правды” по помещениям, группам и датчикам.

Колонки используются так:

B (“Помещение”) → имя пространства (например 316_коридор, 147_вестибюль).

N (или D как fallback) → имя группы света (403_0, 403_1, …).

Важно: D не содержит числа “кол-во групп”, это тоже имя группы, поэтому считать группы “по D как число” нельзя.

G → идентификатор датчика (например 1_19_0 или 1.3.5).

🧱 2) Определение помещения

Пространство “наследуется вниз”: если в строке B пусто, берём значение B из предыдущей строки (ffill).

Номер помещения извлекается из начала строки: ^\d{2,4} (например 403 из 403_кабинет).

Этаж определяется как room // 100 (например 403 → 4 этаж).

🧭 3) Классификация типов помещений

Тип помещения определяется по ключевым словам в B (без строгого формата):

Коридор = “коридор” или “вестибюль”

Лестница = “лестница” или “лестничная”

Кабинет = “кабинет” или “лаб/лаборатория”

Тамбур = “тамбур”

С/у = “с/у/су” или “умывальная”

Если тип/номер определить нельзя — помещение попадает в “пропущенные”.

🔢 4) Количество групп в помещении

Количество групп никогда не берётся из числа в колонках.

group_count считается как:

количество уникальных group_id в помещении,

где group_id = N, а если N пустая — берём D.

🔗 5) Привязка датчиков к группам

Датчики не обязаны идти по порядку.

Правило привязки:

для каждой group_id берём первый встретившийся sensor_id из колонки G;

затем формируем список датчиков в порядке появления групп.

Нормализация sensor_id:

точки заменяем на _: 1.3.5 → 1_3_5,

далее используем паттерны:

sensor.ms_<id>_state

sensor.il_<id>_state

switch.il_<id>_sensor_enable

📚 6) Глоссарий шаблонов — единственный источник вида карточек

Глоссарий — txt-файл со свободным текстом и YAML-вставками карточек.

Граница карточки определяется только строками:

строка {{ отдельно

строка }} отдельно

Regex по {{ ... }} не используется, потому что внутри карточек может быть Jinja {{ ... }} (например в markdown), и это ломает парсинг.

🎯 7) Выбор шаблона карточки

choose_template(type, group_count, templates) работает так:

если есть точный шаблон (type, group_count) — берём его,

иначе берём ближайший по числу групп среди доступных шаблонов этого типа.

Автомасштабирования нет (не добавляем/не удаляем блоки групп внутри карточки).

🧩 8) Подстановки в YAML выполняются “как текстовый компилятор”

Мы не сериализуем YAML, а редактируем текст, чтобы сохранить:

исходные отступы,

формат глоссария,

структуру карточек.

При замене строк сохраняем отступ через захват (\s*), чтобы не ломать YAML.

🏷 9) Общая группа помещения (entity_id)

Для общей группы используем:

light.<room>_obshchii_<tail_slug>

tail_slug строится из “хвоста” имени помещения (после номера) через транслит.

Для спорных слов используется словарь исключений (точечные замены до транслита), например:

вестибюль → vestibiul

умывальная → umyvalnaia

Транслитерация “лестница” правится правилом:

ц → ts, чтобы было lestnitsa, а не lestnica.

🧷 10) Локальные фиксы под особенности шаблонов

Если в шаблоне есть “магический текст”, не являющийся header, например:

name: Тамбур 136

Мы исправляем его контекстно, только рядом с entity общей группы:

для tambour заменяем name: у tile с entity: <general_entity> на Тамбур <room>.

🧾 11) Выходной файл и читаемость

Выход — .txt с карточками, каждая карточка строго обрамлена:

заголовок ### <space_name> (тип=..., групп=..., шаблон=...)

затем {{ ...карточка... }}
Это помогает визуально отделять карточки при копировании в Lovelace.

🧪 12) Логирование и контроль качества

В консоль выводится:

какие шаблоны найдены в глоссарии,

сколько помещений каждого типа прошло фильтры,

какие помещения пропущены (не распознаны / нет шаблона).
