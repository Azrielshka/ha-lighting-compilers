# Принципы, которые не меняются

1. Excel читается один раз, дальше — только parquet.
2. Правила нейминга живут в `scripts/_lib/canon.py`, генераторы их не дублируют.
3. YAML генерируется текстом, а не сериализацией: наладчик читает результат
   глазами и сверяет со своей таблицей.
4. **Каждый шаг запускается отдельно, из CLI, без GUI и без предыдущего шага.**
5. **`Build All` офлайновый.** В живую систему пишет только деплой, по явному
   действию.
6. Launcher — orchestration layer без бизнес-логики.
7. Целевая машина — Windows наладчика. Это не интеграция HA, а внешний
   генератор конфигураций с опциональным деплоем.

# Пайплайн (8 шагов, работает)

```
Excel
  │
  ├─ 1. validate_excel.py          проверка таблицы; на ошибках пайплайн стоит
  ├─ 2. normalize_excel.py         → devices / groups / spaces / units .parquet
  │
  ├─ 3. generate_lights_groups.py  → lights_group.yaml
  ├─ 4. generate_general_groups.py → lights_general_groups.yaml
  ├─ 5. generate_floor_groups.py   → lights_floor_groups.yaml
  ├─ 6. generate_areas.py          → areas.yaml            (Areas + Floors)
  ├─ 7. generate_scripts.py        → scripts.yaml          (клоны шаблонов)
  └─ 8. generate_automations.py    → automations.yaml + blueprints/
```

Не в пайплайне: `show_normalized.py` (просмотр parquet),
`check_file_editor.py` (разведка транспорта деплоя).

## Этап 2 — `validate_excel.py` ✅

Первый шаг. Ничего не пишет в `normalized/`, печатает отчёт и
`data/validation_report.json`. Коды возврата: `0` — можно генерировать,
`1` — блокирующие ошибки (или предупреждения при `--strict`), `2` — файл не найден.

**Блокирующие ошибки:**

| Код | Правило |
|---|---|
| `E01` | нет листа «Проектная БД» |
| `E02` | нет обязательной колонки |
| `E03` | адрес лампы не в формате `X.Y.Z` (или `None` в колонке «Лампа») |
| `E04` | дубликат адреса лампы |
| `E05` | дубликат адреса датчика |
| `E06` | дубликат адреса панели |
| `E07` | один датчик привязан к двум группам (поверх `E05`) |
| `E08` | группа без ламп |
| `E09` | помещение без ламп / без групп |
| `E10` | в помещении колонка «Датчик» пуста во всех строках (нужен адрес или `None`) |
| `E11` | то же для «Панель» |
| `E12` | устройство в строке без группы |
| `E13` | `group_id` встречается в двух помещениях |
| `E14` | нераспознанное значение в «Датчик»/«Панель» |
| `E15` | блок смешивает разные семейства |
| `E16` | блок задан у помещения, которое не автоматизируется |

**Предупреждения:**

| Код | Правило |
|---|---|
| `W01` | помещение без типа → исключается из карточек |
| `W02` | неизвестный тип помещения |
| `W03` | этаж в адресе ≠ колонка «Этаж» (истина — в адресе) |
| `W04` | шина в адресе ≠ колонка «Шина DALI» |
| `W05` | группа без датчиков (кроме `zal`) |
| `W06` | пустая строка внутри листа |
| `W07` | в единице обслуживания больше 12 датчиков |
| `W08` | автоматизируемое помещение без датчиков |

## Этап 3 — `normalize_excel.py` ✅

Длинный формат: **строка = одно устройство**. `device_rows.parquet` упразднён.

**Решение:** normalize отказывается работать на таблице с блокирующими ошибками
(зовёт ту же `validate()`), обход — `--force`. Иначе пайплайн не падает, а тихо
врёт — ровно та проблема, из-за которой затевался переезд.

### 3.1 Жёсткая схема parquet ✅

`pa.Table.from_pandas()` выводил типы из данных: на объекте без панелей
`panels_by_group` становился `list<list<null>>`. Схема файла зависела от объекта,
и генератор сломался бы не у разработчика, а у наладчика. Теперь схема
объявлена явно в `schemas.py`, запись идёт строго по ней.

### 3.2 `show_normalized.py` ✅

Parquet бинарный, глазами не открыть. Режимы: обзор, `--space`, `--group`, `--full`.

## Этап 4 — группы света ✅

Три генератора на новую схему. Фильтры переехали из констант в исходнике в CLI
(`_lib/filters.py`): раньше, чтобы исключить этаж, наладчику пришлось бы править
код, а лаунчер передать их не мог.

**Исправлен дефект v1: висячие сущности.** У `generate_general_groups.py` стояло
`EXCLUDE_SPACE_CONTAINS = ["koridor"]` — артефакт отладки перед ПНР. Коридоры не
получали общую группу, но `generate_floor_groups.py` всё равно тянул их
`general_light_entity` в группу этажа: в HA появлялась группа со ссылкой на
несуществующую сущность.

`TECHNICAL_SPACE_TYPES = {korridor, special, recreation}` — проходные
пространства. `tex_floor_<n>` создаются по умолчанию, отключаются
`--no-tech-groups`.

## Этап 5 — Areas и Floors ✅

Основано на `Auto-area-HA`, который уже читал наш `spaces.parquet`. Разрезан
надвое: `transform.py` → наш генератор, `ha_client.py` → транспорт деплоя.

**Выброшен `TRANSLATION_DICT`:** он восстанавливал русское имя из транслита по
словарю (`103_vestibiul` → `103_Vestibiul`, латиница). Костыль под таблицу v1,
где `space` выглядел как `403_кабинет_медиц`. В v3 колонка «Название помещения»
уже человеческая — имя Area берём прямо из неё, транслит уходит в `aliases`.

Шаг офлайновый: тест `test_generator_makes_no_network_calls` это стережёт.

## Этап 6 — единицы обслуживания ✅

Основание: `uncledrew/logic-contol-system/` — схема
`Датчик → Blueprint → Zone Manager → Скрипт`. Blueprint'ы и скрипты статичны;
генерировать надо клоны скриптов и автоматизации под них.

**Колонка «Блок».** Единица обслуживания = помещение (если блок пуст) либо все
помещения одного блока. Правила заказчика («лестницы по стояку», «санузлы с
ближайшими») опираются на пространственную информацию, которой в таблице нет:
эвристика работала бы на одном объекте и молча врала на другом. Решение
принимает проектировщик, глядя на план.

Семейства: `korridor`/`recreation` → `default`, `hall` → `hall`,
`special` → `special`, `class`/`zal` → не автоматизируются.

## Этап 7 — скрипты ✅

`generate_scripts.py` клонирует шаблоны из `templates/scripts/` по единицам.

**Зачем клонировать:** один экземпляр скрипта в HA — это одна очередь. При
тысяче датчиков вызовы копятся, и свет отстаёт от человека; `mode: queued` не
спасает. У каждой единицы свой набор — они друг друга не ждут.

Меняются ровно две вещи: корневой ключ (из него HA делает entity_id) и `alias`
(иначе в UI было бы двести одинаковых «Шаблон скрипта»). Тело не трогаем —
имён в нём нет, конфигурацию скрипт получает параметрами.

## Этап 8 — автоматизации ✅

`generate_automations.py` — экземпляры blueprint'ов, по две на единицу (ON и OFF).
Плюс копии blueprint'ов в `data/blueprints/` для деплоя.

**Пакет, а не `automations.yaml`:** туда HA пишет автоматизации, созданные через
UI, и перезапись стёрла бы ручную работу наладчика.

**У `special` OFF вход один** (`off_script`), у `default` и `hall` — два.
Имена входов описаны явно в `BLUEPRINT_INPUTS_BY_FAMILY` и сверяются тестом с
самими blueprint'ами: опечатка не уронила бы генерацию — HA молча
проигнорировал бы вход, и автоматизация осталась бы без скрипта.

## Деплой

**Одна кнопка `Deploy` с галочками:** что отправлять. Сеть трогается только здесь.

Транспорт выяснен:

| Что | Куда | Чем |
|---|---|---|
| Areas, Floors | реестры HA | **WebSocket API** (обкатан в `Auto-area-HA`) |
| YAML-файлы | `/config` | **SCP через Advanced SSH add-on** |

**File Editor не годится:** прокси `/api/hassio/` в HA работает по белому
списку, `ingress/session` и `addons/*/info` в него не входят → 401 при любом
токене (проверено `check_file_editor.py` на живом HA).

Особенности SSH-транспорта (`uncledrew/ssh_connection_cooldrew.md`):
SFTP не работает → нужен `scp -O`; обязательны `Ciphers=aes256-ctr` и
`MACs=hmac-sha2-256-etm@openssh.com`, иначе `Corrupted MAC on input`;
порт и юзер свои на каждом объекте → поля в лаунчере, не константы.

# ⚠ Известный долг: система собрана, но не заработает

Конфигурация полная, но на объекте свет работать **не будет**, пока не появятся
две вещи. Обе — вне пайплайна по решению владельца.

| Чего нет | Что произойдёт |
|---|---|
| `input_number.vacant_delay` | OFF-триггер: `for: seconds: {{ states(...) }}` вернёт `unknown` → **свет не будет гаснуть** |
| JSON Zone Manager | Автоматизация вызовет `zone_manager.get_sensor_config`, получит `found: false` → **свет не будет включаться** |

`input_number` — один на объект, отдельным шагом позже.
JSON Zone Manager — собирается вручную; для автоматизации понадобился бы лист
«Группы соседей», который пока пуст.

**Не чиним по решению владельца:** коллизия `room_slug` — два разных названия
помещений могут дать один slug (`201_Зал` и `201_зал` → `201_zal`), и в HA
появятся две группы с одинаковым `unique_id`.
