# Принципы, которые не меняются

1. Excel читается один раз, дальше — только parquet.
2. Правила нейминга живут в `scripts/_lib/canon.py`, генераторы их не дублируют.
3. YAML генерируется текстом, а не сериализацией.
4. **Каждый шаг запускается отдельно, из CLI, без GUI и без предыдущего шага.**
   Генераторы работают на том, что уже лежит в `data/normalized/`.
5. Launcher — orchestration layer без бизнес-логики.
6. Целевая машина — Windows наладчика. Это не интеграция HA, а внешний
   генератор конфигураций с опциональным деплоем.

## Этап 2 — `scripts/validate_excel.py` (новый шаг) ✅

Отдельный первый шаг пайплайна. Читает Excel, ничего не пишет в `normalized/`,
печатает отчёт и возвращает код возврата. Пишет `data/validation_report.json`.

Коды возврата: `0` — можно генерировать, `1` — блокирующие ошибки
(или предупреждения при `--strict`), `2` — файл не найден.

**Блокирующие ошибки (exit code ≠ 0):**

| Код | Правило |
|---|---|
| `E01` | нет листа «Проектная БД» |
| `E02` | нет обязательной колонки |
| `E03` | адрес не в формате `X.Y.Z` |
| `E04` | дубликат адреса лампы в пределах объекта |
| `E05` | дубликат адреса датчика |
| `E06` | дубликат адреса панели |
| `E07` | один датчик привязан к двум группам |
| `E08` | группа без ламп |
| `E09` | помещение без ламп / без групп |
| `E10` | в помещении нет ни одного значения в «Датчик» (ни адреса, ни `None`) |
| `E11` | то же для «Панель» |
| `E12` | устройство в строке без группы (привязать его не к чему) |
| `E13` | `group_id` встречается в двух разных помещениях |
| `E14` | нераспознанный токен в «Датчик»/«Панель» |

`E05` срабатывает на любой дубликат адреса датчика; `E07` добавляется сверху,
когда дубликат ещё и разъехался по разным группам.
`None` в колонке «Лампа» — это `E03`: строка существует ради лампы.

**Предупреждения (exit code 0, генерация идёт):**

| Код | Правило |
|---|---|
| `W01` | помещение без типа → исключается из карточек |
| `W02` | неизвестный тип помещения |
| `W03` | `X` в адресе ≠ колонка «Этаж» |
| `W04` | `Y` в адресе ≠ колонка «Шина DALI» |
| `W05` | группа без датчиков при типе, отличном от `zal` |
| `W06` | пустая строка внутри листа |

- [x] флаг `--strict` превращает предупреждения в ошибки
- [x] тесты на синтетических таблицах: по одному кейсу на правило (`tests/`, 69 тестов)

**Решение по ходу:** normalize отказывается работать на таблице с блокирующими

независимыми — `validate_excel.py` по-прежнему запускается отдельно.

### Что сделано сверх переноса

**Фильтры переехали в CLI** (`scripts/_lib/filters.py`). Раньше они были
константами в исходнике: чтобы исключить этаж, наладчику пришлось бы править
код, а лаунчер передать их не мог вообще.

```
--spaces 103_vestibiul 104_metodicheskii_kabinet   # или русским именем
--floors 1 2                                       # только floor
--exclude-floors 4
--exclude-space-contains sklad server
--no-tech-groups                                   # только floor
```

`--spaces` и `--exclude-space-contains` понимают и транслит, и русское имя:
наладчику неочевидно, что в фильтр надо писать `103_vestibiul`.

**Исправлен дефект v1: висячие сущности.** У `generate_general_groups.py`
стояло `EXCLUDE_SPACE_CONTAINS = ["koridor"]` — артефакт отладки перед ПНР.
Коридоры не получали общую группу, но `generate_floor_groups.py` всё равно
тянул их `general_light_entity` в группу этажа: в HA появлялась группа,
ссылающаяся на несуществующую сущность. Теперь общую группу получают все
помещения; тест `test_no_dangling_entities` проверяет, что каждая ссылка
ведёт на реально созданную сущность.

**Общий рендер YAML** (`scripts/_lib/yaml_render.py`): три генератора писали
один и тот же блок `platform: group` тремя копиями кода.

**Порядок — как в таблице** во всех трёх (раньше `floor` сортировал entity_id
по алфавиту). Наладчик сверяет YAML со своим Excel.

**Помещение без типа** попадает во все три генератора: лампы в нём физически
существуют. Отсекается оно только на карточках.
