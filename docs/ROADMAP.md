# Принципы, которые не меняются

1. Excel читается один раз, дальше — только parquet.
2. Правила нейминга живут в `scripts/_lib/canon.py`, генераторы их не дублируют.
3. YAML генерируется текстом, а не сериализацией: наладчик читает результат
   глазами и сверяет со своей таблицей.
4. **Каждый шаг запускается отдельно, из CLI, без GUI и без предыдущего шага.**
5. **`Build All` офлайновый.** В живую систему пишет только деплой, по явному
   действию.
6. Launcher — orchestration layer без бизнес-логики.
7. Целевая машина — Windows наладчика. Это не интеграция HA, а внешний
   генератор конфигураций с опциональным деплоем.

# Пайплайн (9 шагов, работает)

```
Excel
  │
  ├─ 1. validate_excel.py          проверка таблицы; на ошибках пайплайн стоит
  ├─ 2. normalize_excel.py         → devices / groups / spaces / units .parquet
  │
  ├─ 3. generate_lights_groups.py  → lights_group.yaml
  ├─ 4. generate_general_groups.py → lights_general_groups.yaml
  ├─ 5. generate_floor_groups.py   → lights_floor_groups.yaml
  ├─ 6. generate_areas.py          → areas.yaml            (Areas + Floors)
  ├─ 7. generate_helpers.py        → lighting-compilers.yaml (помощники)
  ├─ 8. generate_scripts.py        → scripts.yaml          (клоны шаблонов)
  └─ 9. generate_automations.py    → automations.yaml + blueprints/
```

Не в пайплайне: `generate_lovelace_cards.py` (карточки → views дашборда),
`show_normalized.py` (просмотр parquet), `check_sftp.py` (разведка канала
деплоя), `backup_dashboard.py` (снимок дашборда перед деплоем карточек).

# Осталось

## Отложено владельцем

- [ ] **UI блокируется на время выполнения.** `subprocess.run()` держит поток,
      окно не отвечает: Build All на тестовой таблице — 8.7 с, на объекте в
      76 помещений порядка 12–15 с, на 12 этажей до полуминуты. Windows
      показывает «Не отвечает».

      ⚠ Замер вскрыл вторую сторону: почти всё это время — **не работа, а
      холодный старт Python**. Каждый из девяти шагов тратит ~950 мс на импорт
      интерпретатора и pandas, сама генерация занимает миллисекунды. Лечится
      двумя независимыми способами: вынести выполнение из GUI-потока
      (QThread/QProcess — окно перестанет висеть) и прогонять пайплайн одним
      процессом (уберёт ~8 с из 8.7). Второе выгоднее: оно убирает ожидание,
      а не показывает его красиво.

      ⚠ Есть и вторая, менее очевидная сторона — она была записана в
      `know-limitation.md` ещё в марте и с тех пор не изменилась: кнопки
      выглядят выключенными, но **клики по ним копятся в очереди Qt** и
      срабатывают, когда синхронный шаг закончится. То есть наладчик, ткнув
      «Build All» дважды на подвисшем окне, получит второй прогон. Лечится тем
      же переносом в отдельный поток.

      Отложено 2026-07-18.

# ⚠ Известный долг: система собрана, но не заработает

Конфигурация полная, но на объекте свет **не включится**, пока не появится
JSON Zone Manager.

| Чего нет | Что произойдёт |
|---|---|
| JSON Zone Manager | Автоматизация вызовет `zone_manager.get_sensor_config`, получит `found: false` → **свет не будет включаться** |

Собирается вручную; для автоматизации понадобился бы лист «Группы соседей»,
который пока пуст. Помощником не является — это отдельная конфигурация, шагом
helpers не закрывается.

**`input_number.vacant_delay` из этого списка ушёл** (2026-07-17): его создаёт
`generate_helpers.py` со значением по умолчанию. Долг «свет не гаснет» закрыт.

**Не чиним по решению владельца:** коллизия `room_slug` — два разных названия
помещений могут дать один slug (`201_Зал` и `201_зал` → `201_zal`), и в HA
появятся две группы с одинаковым `unique_id`.
